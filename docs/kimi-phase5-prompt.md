## CRITICAL GOTCHAS — verified against the real source; do not rediscover

**1. The ident-naming transform (HIGHEST FRICTION — v1 has NONE of it).** v1 emits YANG names VERBATIM as Rust idents (`pub {}: {}` with `child.name()`, lib.rs:73, 142) and only splits on `-_.` for PascalCase (`to_pascal_case`, lib.rs:319). There is NO reserved-word check, NO leading-digit guard, NO collision disambiguation. This breaks on real schemas:
- YANG `leaf type {...}` / `match` / `loop` / `struct` / `fn` / `ref` → `pub type: String` = uncompilable Rust.
- YANG `2nd-addr` → `pub 2nd_addr` = uncompilable.
- two YANG names `foo-bar` and `foo_bar` both → `foo_bar` = silent clobber.

**SAFE PATTERN:** define a deterministic transform. Field idents: hyphen→underscore, snake_case; if the result is a Rust keyword, emit `r#kw` (or for the strict keywords `crate`/`self`/`super`/`Self` that cannot be raw, suffix `_`); if it starts with a digit, prefix `_`; detect collisions (two YANG names → one ident) and disambiguate deterministically (`_2`, `_3` in declaration order). Type idents: CamelCase via the existing `to_pascal_case`. **The XML/JSON WIRE name MUST stay the original YANG name — the manifest stores YANG names; only the Rust ident is escaped. Keep field-ident and wire-name as SEPARATE strings** (the serializer emits the wire name; the struct field uses the ident). Assert the final type/variant/field set is unique before emit; fail generation loudly rather than emit clobbered code.

**3. RFC-7951 JSON_IETF encoding the native serializer MUST match byte-for-byte (verified empirically against the vendored libyang):**
- **Scalar encoding is NON-uniform.** `int64`/`uint64`/`decimal64` are QUOTED STRINGS (`"9223372036854775807"`, `"3.14"`); `int8..int32`/`uint8..uint32` are BARE NUMBERS (`-5`); `boolean` is a bare literal (`true`); `enum`/`string`/`identityref` are strings; leaf-list → JSON array. Branch the emitted JSON serializer on base type per field at generate time. A bare `format!("{}", self.big)` for an `i64` emits a bare number → byte-gate FAIL.
- **Module prefix only at a namespace boundary (RFC-7951 §4).** Top-level node is `"<module>:<name>"`; nested children in the SAME module are UNprefixed; an augment-from-another-module child re-qualifies. Compute prefix-vs-no-prefix per field at generate time via `SchemaNodeRef::module()` vs the parent's module. Use `Module::name()` (schema.rs:782) for the JSON `module:` qualifier (NOT the YANG `prefix` token) and `Module::namespace()` (787) for XML `xmlns`.
